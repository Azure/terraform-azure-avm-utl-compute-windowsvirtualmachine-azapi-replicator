# Task #26 - os_managed_disk_id

## Summary

Implemented the `os_managed_disk_id` field as an optional root-level argument that enables attaching an existing managed disk to a VM. When set, the VM uses `CreateOption = "Attach"` and omits the `osProfile`; when not set, the VM uses `CreateOption = "FromImage"` with full `osProfile`. All ConflictsWith and ExactlyOneOf validations have been implemented.

## Create Phase Verification

### Pattern Identification

Queried the Create method from the azurerm provider source code:

**Evidence from resourceWindowsVirtualMachineCreate:**

```go
func resourceWindowsVirtualMachineCreate(d *pluginsdk.ResourceData, meta interface{}) error {
	client := meta.(*clients.Client).Compute.VirtualMachinesClient
	// ...
	
	managedDiskIdRaw := d.Get("os_managed_disk_id").(string)
	// Note: The API fails if OsProfile is anything but nil with CreateOption = "Attach"
	osDiskIsImported := managedDiskIdRaw != ""
	
	// ... build params ...
	
	if !osDiskIsImported {
		// CREATE MODE: Build complete OsProfile
		params.Properties.OsProfile = &virtualmachines.OSProfile{
			AdminPassword:            pointer.To(d.Get("admin_password").(string)),
			AdminUsername:            pointer.To(d.Get("admin_username").(string)),
			ComputerName:             pointer.To(computerName),
			AllowExtensionOperations: pointer.To(allowExtensionOperations),
			WindowsConfiguration: &virtualmachines.WindowsConfiguration{
				ProvisionVMAgent:       pointer.To(provisionVMAgent),
				EnableAutomaticUpdates: pointer.To(autoUpdatesEnabled),
				WinRM:                  expandWinRMListener(d.Get("winrm_listener").(*pluginsdk.Set).List()),
			},
			Secrets: secrets,
		}
		// Set image reference for new disk creation
		params.Properties.StorageProfile.ImageReference = expandSourceImageReference(...)
	} else {
		// ATTACH MODE: Use existing disk, NO OsProfile
		diskId, err := commonids.ParseManagedDiskID(managedDiskIdRaw)
		if err != nil {
			return err
		}
		osDisk.ManagedDisk.Id = pointer.To(diskId.ID())
		osDisk.CreateOption = virtualmachines.DiskCreateOptionTypesAttach
	}
	
	// Single-phase create
	if err := client.CreateOrUpdateThenPoll(ctx, id, params, virtualmachines.DefaultCreateOrUpdateOperationOptions()); err != nil {
		return fmt.Errorf("creating Windows %s: %+v", id, err)
	}
}
```

**Pattern:** Single-phase create operation using `CreateOrUpdateThenPoll`.

**Decision:** The field participates in the primary Create phase and controls a fundamental mode switch in the VM creation logic. Implemented in `local.body` with conditional `createOption` and disk ID assignment.

## Special Handling

### 1. Optional + Computed Nature

**Challenge:** The field is both Optional (input) and Computed (output). In azurerm provider:
- When NOT provided by user → VM creates new disk → Field is computed with the created disk's ID
- When provided by user → VM attaches existing disk → Field stores user's input value

**Implementation Decision:**
- The Replicator Module CANNOT replicate the Computed behavior (computing the disk ID after creation)
- This is acceptable because:
  1. The primary use case (CREATE mode with new disk) works without this field
  2. Users who need ATTACH mode can explicitly provide the disk ID
  3. The azurerm provider's Computed behavior is an output convenience, not a functional requirement
  4. Users can query the VM resource after creation to get the disk ID if needed

### 2. ForceNew

**Schema:** `ForceNew: true`

**Implementation:** Added to `replace_triggers_external_values`:

### 6. Conditional Body Logic

**Provider behavior:** When `os_managed_disk_id` is set, the provider:
1. Does NOT create `osProfile`

3. Sets `managedDisk.id` to the provided disk ID

- **ATTACH mode:** User provides `os_managed_disk_id`, does NOT provide `admin_username` → No `osProfile` created
- **CREATE mode:** User provides `admin_username`, does NOT provide `os_managed_disk_id` → `osProfile` created

The validations enforce that these fields cannot be used together, ensuring the correct mode is selected.

## Critical Review & Edge Case Analysis

### Null Semantics
- **null value:** Means "create new disk from image" (FromImage mode)
- **non-null value:** Means "attach existing disk" (Attach mode)
- **Behavior:** Null is the default and common case; non-null enables the alternative attach mode

### Boundary Conditions
- **Empty string:** Not allowed - field is nullable, not an empty string
- **Invalid format:** Caught by format validation regex
- **Non-existent disk ID:** Will fail at Azure API call (acceptable - Azure validates resource existence)

### Idempotency
- **Same disk ID:** Terraform detects no change, no action taken
- **Change disk ID:** Triggers replacement (ForceNew), creates new VM with different disk
- **Safe:** ForceNew ensures we never attempt to "swap" disks on an existing VM

### Safe References
- **Conditional access:** Always check `var.os_managed_disk_id != null` before using
- **Body construction:** Uses conditional merge to only add disk ID when present
- **No nested access:** Field is a simple string, no nested properties to safely access

### Mode Switch Logic
- **ATTACH mode requires:** os_managed_disk_id + NO admin_username + NO other OsProfile fields
- **CREATE mode requires:** admin_username + (source_image_id OR source_image_reference) + NO os_managed_disk_id
- **Enforced by:** ExactlyOneOf and ConflictsWith validations prevent invalid combinations
- **Natural behavior:** The existing conditional osProfile logic (based on admin_username) already handles the mode switch correctly

### Computed Value Limitation
- **Provider behavior:** Field is Computed - when not provided, populated with created disk ID
- **Module limitation:** Cannot replicate Computed behavior (no access to Azure-created disk ID)
- **Impact:** Users cannot reference `module.vm.os_managed_disk_id` in CREATE mode
- **Acceptable because:**
  1. Primary use case (CREATE mode) doesn't need this output
  2. Users can query the VM resource separately to get disk ID if needed
  3. The functional behavior (VM creation) is fully replicated
  4. This is an output convenience, not a functional requirement
