# Task #25 - max_bid_price - Proof Document

## Summary
Implemented the `max_bid_price` argument for `azurerm_windows_virtual_machine` to `azapi_resource`. The field is Optional with a default of -1, validated to be >= -1.0, and can only be set > 0 when priority is "Spot". The provider only sets billingProfile.maxPrice when the value is > 0, following exact provider logic.

- When `max_bid_price = -1` (default) or any value <= 0: `billingProfile` is NOT included in the body
- When `max_bid_price > 0`: `billingProfile.maxPrice` is set with the value
- This exactly replicates the provider's behavior

## Critical Review & Edge Case Analysis

### Null Semantics
- **Value = -1 (default):** Means "don't evict for price reasons". Implemented by NOT setting the field in the API body.
- **Value = 0:** Invalid if we follow strict provider logic (validation checks v > 0). Not set in API body.
- **Value > 0:** Set the specific max price. Field is set in API body.

### Boundary Conditions
- **Minimum value:** -1.0 (validated)
- **Maximum value:** No upper limit in validation
- **Default:** -1
- **Zero:** Allowed by validation but won't trigger API field setting (v > 0 check)

### Idempotency
The conditional logic `var.max_bid_price > 0` ensures:
- When value is -1 (default), the field is consistently omitted
- When value > 0, the field is consistently present with the same value
- No order-dependent behavior

### Safe References
Direct number comparison, no nested access needed. Safe to use directly in conditional.

### Edge Case: Updating from Spot to Regular Priority
If a user changes priority from "Spot" to "Regular" while max_bid_price > 0:
- The validation will fail at plan time (max_bid_price can only be set when priority is Spot)
- User must also change max_bid_price to <= 0
- This matches the provider behavior which would error in Create/Update

### Edge Case: Default Value Behavior
The provider sets Default: -1, which means:
- If user doesn't specify the field, it defaults to -1
- -1 means "don't evict for price reasons" (up to on-demand price)
- The field is NOT sent to the API when value is -1
- This is achieved by the `v > 0` check in provider and replicated with `var.max_bid_price > 0` condition
